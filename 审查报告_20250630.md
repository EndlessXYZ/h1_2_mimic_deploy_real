## 🔴 高风险

### 1. `action_manager` 数据竞争

- **位置:**
  - [State_RLBase.h](file:///home/meme/Documents/unitree_h1/h1_2_mimic_deploy_real/include/FSM/State_RLBase.h) L58 (`run()` 读取)
  - [state_RLBase 策略线程 enter()](file:///home/meme/Documents/unitree_h1/h1_2_mimic_deploy_real/include/FSM/State_RLBase.h) `policy_thread` lambda 内 `env->step()` 调用 `process_action()`
  - [State_Mimic.cpp](file:///home/meme/Documents/unitree_h1/h1_2_mimic_deploy_real/robots/h1_2/src/State_Mimic.cpp) L344-346 (Mimic 的 `run()` 也读取 processed_actions)
- **问题:** 策略线程 (50Hz) 调用 `action_manager->process_action()` 写入 `processed_actions`，FSM 主线程 (1kHz) 调用 `run()` 在其 `registered_checks` 中读取 `processed_actions()`。两者无锁、无原子变量、无内存屏障。
- **影响:** FSM 线程可能读到半写状态的动作向量，导致电机命令在帧间跳变。
- **缓解因素:** 50Hz 写一次后 FSM 会重复读同一值到下次 step()，实际跳变概率较低，但无硬件保证。
- **建议修复方向:** 对 `processed_actions` 的读写加锁，或使用 `std::atomic` 包装，或在策略线程与 FSM 主线程间使用 double buffer。

### 2. `episode_length` / `global_phase` 非原子访问

- **位置:** [manager_based_rl_env.h](file:///home/meme/Documents/unitree_h1/h1_2_mimic_deploy_real/include/isaaclab/envs/manager_based_rl_env.h) L87-88
  ```cpp
  long episode_length = 0;
  float global_phase = 0.0f;
  ```
- **问题:** `episode_length` (`long`, 64-bit) 和 `global_phase` (`float`) 被策略线程在 `step()` 中读写 (++`episode_length`)，同时被 FSM 线程的 `registered_checks` lambda 通过 `env_ptr->episode_length` 读取。
- **影响:** 64-bit `long` 在 32-bit 平台上非原子写入，FSM 线程可能读到撕裂值（高/低 32-bit 来自不同时刻）。
- **建议修复方向:** 使用 `std::atomic<long>` 和 `std::atomic<float>`，或在读取侧使用 relaxed load 配合 write-side release-store。

## 🟡 中风险

### 3. `run()` 与 `enter()` 中 lowcmd 索引方式不一致

- **位置:**
  - `run()`: [State_Mimic.cpp](file:///home/meme/Documents/unitree_h1/h1_2_mimic_deploy_real/robots/h1_2/src/State_Mimic.cpp) L345-346 vs [State_RLBase.cpp](file:///home/meme/Documents/unitree_h1/h1_2_mimic_deploy_real/robots/h1_2/src/State_RLBase.cpp) L51-52
  - `enter()`: [State_Mimic.cpp](file:///home/meme/Documents/unitree_h1/h1_2_mimic_deploy_real/robots/h1_2/src/State_Mimic.cpp) L216-217 vs [State_RLBase.h](file:///home/meme/Documents/unitree_h1/h1_2_mimic_deploy_real/include/FSM/State_RLBase.h) L20-23
- **问题:**
  - Mimic 使用 `joint_ids_map[i]` 索引:
    ```cpp
    lowcmd->msg_.motor_cmd()[env->robot->data.joint_ids_map[i]].q() = action[i];
    ```
  - WalkVelocity 使用 `i` 直接索引:
    ```cpp
    lowcmd->msg_.motor_cmd()[i].q() = action[i];
    ```
  - 当前 `joint_ids_map` 是恒等映射 `[0,1,2,...,26]`，两者等价。
  - 但如果未来修改 `joint_ids_map`（重排电机通道或只控制部分关节），WalkVelocity 的索引方式会写错 channel。
  - 同样问题也存在于 `enter()` 中设置 PD 增益时。
- **建议修复方向:** 统一 WalkVelocity (`State_RLBase`) 也使用 `joint_ids_map[i]` 索引。

### 4. `State_Mimic::motion` static shared_ptr 不重置

- **位置:**
  - 声明: [State_Mimic.h](file:///home/meme/Documents/unitree_h1/h1_2_mimic_deploy_real/robots/h1_2/include/State_Mimic.h) L33
    ```cpp
    static std::shared_ptr<MotionLoader_> motion; // for obs computation
    ```
  - 赋值: [State_Mimic.cpp](file:///home/meme/Documents/unitree_h1/h1_2_mimic_deploy_real/robots/h1_2/src/State_Mimic.cpp) L222
    ```cpp
    motion = motion_; // set for specific motion
    ```
  - exit(): [State_Mimic.h](file:///home/meme/Documents/unitree_h1/h1_2_mimic_deploy_real/robots/h1_2/include/State_Mimic.h) L22-29 仅 join 线程 + 关文件，未重置 `motion`
- **问题:** `exit()` 未将 `motion` 重置为 `nullptr`。切换到其他状态后，旧 `MotionLoader_` 对象引用计数不会归零，内存持续占用。此外，`motion_command` / `motion_anchor_ori_b` 两个全局注册的 observation lambda 可能在其他环境下被调用，读到过期数据。
- **建议修复方向:** 在 `exit()` 中添加 `motion.reset();`。

### 5. `registered_checks` lambda 的 `env_ptr` 悬空风险

- **位置:** [State_Mimic.cpp](file:///home/meme/Documents/unitree_h1/h1_2_mimic_deploy_real/robots/h1_2/src/State_Mimic.cpp) L166-174
  ```cpp
  auto* env_ptr = env.get();
  this->registered_checks.emplace_back(
      std::make_pair(
          [env_ptr, step_dt, time_end]()->bool{ return (env_ptr->episode_length * step_dt) > time_end; },
          end_state_id
      )
  );
  ```
- **问题:** lambda 按值捕获了裸指针 `env_ptr`。若 FSM 框架未来销毁旧 `State_Mimic` 对象 (释放 `env` unique_ptr)，`env_ptr` 变为悬空指针，再次调用此 lambda 导致 use-after-free。
- **建议修复方向:** 将超时检测改为由 CtlrFSM 在状态切换后自动注销此 check，或将 `env` 改为 `shared_ptr` 由 lambda 持有引用。

### 6. `dump_file_` 异常安全

- **位置:** [State_Mimic.cpp](file:///home/meme/Documents/unitree_h1/h1_2_mimic_deploy_real/robots/h1_2/src/State_Mimic.cpp) L200-209
- **问题:** `enter()` 中打开 dump 文件后，后续代码（设置电机增益、启动线程）若抛出异常，`dump_file_` 不会关闭。`ofstream` 析构函数虽然会关闭，但 `State_Mimic` 对象可能未被立即销毁（由 FSM 以 `shared_ptr` 持有）。
- **建议修复方向:** 使用 RAII guard 包装文件打开操作，或在 L200 附近添加 `try-finally` / scope_exit 保护。

### 7. `step_dt` 等价性不确定

- **位置:** [State_Mimic.cpp](file:///home/meme/Documents/unitree_h1/h1_2_mimic_deploy_real/robots/h1_2/src/State_Mimic.cpp) L159
  ```cpp
  env->step_dt = motion_->dt;  // motion_->dt = 1.0f / fps
  ```
- **问题:** Mimic 覆盖 `step_dt = 1/fps`，WalkVelocity 使用 deploy.yaml 的固定值 `0.02`。若 `.npz` 的 `fps ≠ 50`，两个状态的步长不同。需确认 `dance1_subject2.npz` 的实际 fps。
- **建议修复方向:** 确认并注释 `.npz` 文件的实际 fps；若预期 fps≠50，需确认两个状态步长差异对切换的影响。

### 8. 数组长度不匹配风险 ({run()})

- **位置:**
  - [State_Mimic.cpp](file:///home/meme/Documents/unitree_h1/h1_2_mimic_deploy_real/robots/h1_2/src/State_Mimic.cpp) L344-348
    ```cpp
    auto action = env->action_manager->processed_actions();
    for(int i(0); i < env->robot->data.joint_ids_map.size(); i++) {
        lowcmd->msg_.motor_cmd()[env->robot->data.joint_ids_map[i]].q() = action[i];
    }
    ```
  - [State_RLBase.cpp](file:///home/meme/Documents/unitree_h1/h1_2_mimic_deploy_real/robots/h1_2/src/State_RLBase.cpp) L50-53
    ```cpp
    auto action = env->action_manager->processed_actions();
    for(int i(0); i < action.size(); i++) {
        lowcmd->msg_.motor_cmd()[i].q() = action[i];
    }
    ```
- **问题:** `action.size()` 与 `joint_ids_map.size()` 没有显式长度校验。若 ONNX 输出维度 ≠ 27，或 `joint_ids_map` 被修改，Mimic 的循环以 `joint_ids_map.size()` 为界，WalkVelocity 以 `action.size()` 为界，行为不一致。
- **建议修复方向:** 添加 `assert(action.size() == joint_ids_map.size())` 或运行时 min 保护。

### 9. CtrlFSM 构造函数异常 = 直接崩溃

- **位置:** [CtrlFSM.h](file:///home/meme/Documents/unitree_h1/h1_2_mimic_deploy_real/include/FSM/CtrlFSM.h) L26-51
- **问题:** `YAML::LoadFile`、`getFsmMap().find`、`fsm_class->second()` 等操作抛异常时无 catch，直接 `std::terminate()`。特别是 ONNX 加载失败需要优雅提示。
- **建议修复方向:** 在 `CtrlFSM` 构造和 `state->enter()` 路径添加 `try-catch`，记录日志后回退到 Passive 状态。

### 10. `bad_orientation` 回退后 Passive 的 KD 可能不足

- **位置:** [config.yaml](file:///home/meme/Documents/unitree_h1/h1_2_mimic_deploy_real/robots/h1_2/config/config.yaml) L26-32
- **问题:** 摔倒检测触发后切到 Passive，Passive 的 `kd` 最大只有 6。若机器人在非站立姿态摔倒，此阻尼可能不足以安全支撑。
- **建议修复方向:** 评估 Passive 在非站立姿态下的阻尼需求，或切换到 FixStand 作为中间状态。
